## 一、願景宣言

**智研 AI 法律作業系統（Zhiyan AI Legal OS）** 是一個以台灣法律體系為核心的 AI 原生平台。我們的目標不是打造另一個聊天機器人，而是建立一套**可組合、可驗證、可擴展的法律 AI 基礎設施**——讓不同 AI 代理、提示工程框架、知識庫與前端介面，都能在同一個法律語意層上協作。

```
┌─────────────────────────────────────────────────────────┐
│               Zhiyan AI Legal Operating System           │
├──────────────┬──────────────┬──────────────┬──────────────┤
│  AI Agents   │ Prompt Eng.  │ Knowledge    │ Frontend     │
│  OpenClaw    │ SRP-CORE     │ RAG (FTS5)   │ PWA (Flutter)│
│  Claude Code │ TYPE-S       │ Judicial API │ REST API     │
│  Gemini CLI  │ Persona      │ Case DB      │ Docker       │
│  GPT         │ Citation v2  │ Statute Lib  │ CLI          │
└──────────────┴──────────────┴──────────────┴──────────────┘
         ↓                    ↓                    ↓
   ┌─────────────────────────────────────────────────────┐
   │          Unified Law Semantics Layer (ULSL)          │
   │  強制引用 · 安全路由 · 事實分級 · 多模型路由        │
   └─────────────────────────────────────────────────────┘
```

## 二、核心問題：台灣法律 AI 的碎片化困境

當前台灣法律 AI 生態面臨四大斷裂：

| 問題 | 描述 | 影響 |
|:-----|:------|:------|
| **模型碎片化** | GPT、Claude、Gemini 各自為政，提示詞無法互通 | 重複開發，品質不一 |
| **知識孤島** | 法條資料散落各處，缺乏統一檢索層 | 每個專案重新建置知識庫 |
| **引用不可驗證** | AI 生成法律內容經常捏造法條與判決 | 無法用於實務，存在風險 |
| **缺乏標準架構** | 無統一的安全路由、事實閘門、人格切換機制 | 難以規模化與維護 |

**智研 AI 法律作業系統的使命**：用一套分層架構，解決上述所有問題。

## 四、平台元件生態系

### 4.1 AI 代理（Agents）

智研法律作業系統**不綁定特定模型或代理**，而是提供一套統一的法律語意層，讓以下代理都能接入：

| 代理 | 角色 | 整合方式 |
|:-----|:------|:----------|
| **OpenClaw** | 法律研究專用代理 | 透過 Hermes Agent skill 呼叫，繼承 SRP + Citation v2.1 |
| **Claude Code** | 合約審查 & 程式碼層級法律邏輯 | 透過 API runner + TYPE-S 型態系統 |
| **Gemini CLI** | 快速法條查詢 & 概念釐清 | 透過 OpenAI-compatible API 介面 |
| **GPT (OpenAI)** | 一般法律諮詢 & 書狀生成 | 完整系統提示載入，七層架構全支援 |

**關鍵設計原則**：所有代理共享相同的：
- 安全路由協議（SRP）
- 引用政策（Citation Policy v2.1）
- 事實分級閘門（CORE_GATE）
- 人格切換系統（Persona Layer）
- RAG 知識庫（LOCAL_RAG + Judicial API）

### 4.2 提示工程（Prompt Engineering）

90+ 份規格文件構成完整的提示工程體系：

```
10_核心控制層/       G0, SRP, CORE_GATE, TYPE_S, 子代理
20_模式與引用層/     MODE_ROUTER, CITATION_POLICY
30_安全與治理層/     風險評分, 審計日誌
40_模組與人格層/     6 人格 + 訴訟推演 + 合約風險
50_參考輔助層/       閱卷標準, 常用書狀
60_概念詞條/         法律名詞, 事實摘要, 衝突檢查
70_評估指標層/       不虛構率, 引用準確率, 安全性
80_封存參考/         歷史版本與演進記錄
90_維運治理/         冒煙測試, 功能驗收, 匯出對照
```

### 4.3 法律知識庫（Knowledge Base）

| 元件 | 內容 | 規模 |
|:-----|:------|:------|
| **LOCAL_RAG** | 法條白話摘要 + SQLite FTS5 | 47,001 條（每日同步） |
| **Judicial API** | 司法院判決書查詢 + 裁判書 API | ≥ 100 萬筆判決書 |
| **Case Verify** | 判決書驗證 + 律師事務所實務見解 | 即時查詢 |
| **Statute Lib** | 六法全書分類索引 | 完整台灣法規體系 |

### 4.4 RAG 檢索增強生成

智研 RAG 層的特色：

- **雙源檢索**：LOCAL_RAG（白話摘要）→ Judicial API（原始判決書）
- **FTS5 全文搜尋**：支援中文分詞 + 同義詞擴展
- **強制引用**：任何非 RAG 檢索到的資訊必須標示「待查」或「推論」
- **信心分數**：每條檢索結果附帶信心排名

### 4.5 PWA 前端

Flutter 建構的跨平台 PWA（`law_monitor_app/`）：

```
智研法律工作站 PWA
├── 對話介面（Chat UI）
├── 法條查詢（Statute Browser）
├── 判決書搜尋（Case Search）
├── 書狀生成（Document Generator）
├── 訴訟推演（Litigation Simulator）
└── 系統管理（Admin Dashboard）
```

## 五、技術哲學

### 5.1 設計原則

| 原則 | 說明 | 實踐 |
|:------|:------|:------|
| **信心優先** | 輸出前先宣告信心，低信心直接中止 | G0 Confidence-first |
| **可驗證性** | 每條法律引用必須可回溯原始來源 | Citation Policy v2.1 |
| **安全預設** | 風險評估先於法律分析 | SRP (RL0–RL3) |
| **分層解耦** | 每層獨立可測試、可替換 | 7-Layer Architecture |
| **模型無關** | 不綁定 API 供應商，可隨意切換模型 | OpenAI-compatible runner |
| **可重現性** | 研究可完整重現，支援消融實驗 | `--dry-run` + test suite |

### 5.2 為何不是一般聊天機器人

```
一般聊天機器人         →   智研法律作業系統
──────────────              ──────────────────
單一模型                   多模型協作
無引用                     強制引用 v2.1
無安全過濾                 SRP 分層安全路由
無事實檢查                 CORE_GATE 事實閘門
無法切換人格               6 種專業人格
無知識庫                   RAG + Judicial API
封閉系統                   開放架構 + 插件生態
```
