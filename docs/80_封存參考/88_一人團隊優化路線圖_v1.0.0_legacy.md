## 🚀 建議執行順序（一人工作量優先）

### **第 1 週：快速贏（Low-Hanging Fruit）**

#### 【第 1 天】① 來源信度分數 ✅ 立即做

**工作量**：⭐ 1～2 小時

**步驟**：

```
Step 1: 設計信度分數規則（GPT 協助）
  ├─ VERIFIED = 0.95
  ├─ CROSS_REFERENCED = 0.75
  ├─ USER_REPORTED = 0.30
  └─ NEED_CHECK = 0.10

Step 2: 在 GPT 上測試
  └─ 提示詞：「這份法律文件怎麼評信度？」
     → 看 GPT 怎麼應用

Step 3: 製作「信度評估簡表」
  └─ 放進 Perplexity CORE_GATE

Step 4: 更新文檔
  └─ 在 05_CITATION_POLICY_v2.0.md 加一段說明
```

**產出物**：

```markdown
### 【新增】來源信度評分系統

每個來源自動評分 0～1：

| 來源類型 | 分數 | 使用場景 |
|---------|------|--------|
| VERIFIED | 0.95 | 支撐核心結論 |
| CROSS_REFERENCED | 0.75 | 輔助參考 |
| USER_REPORTED | 0.30 | 背景資訊 |
| NEED_CHECK | 0.10 | 禁止使用 |

【輸出示例】
根據官方統計，...

 台灣證券交易所官方統計
    來源類型：VERIFIED
    信度分數：0.95/1.0
    最後更新：2025-12-15
```

**實施時間**：1～2 小時  
**收益**：✅ 馬上可在 Perplexity 應用

---

#### 【第 2～3 天】④ JSON 標準化（核心輸出）✅ 優先度最高

**工作量**：⭐ 2～3 小時

**為什麼優先做它？**

```
理由：
- 這是「後續所有優化的基礎」
- 一旦定義好，其他優化都能套用
- GPT 可以直接看懂 JSON，方便協作
- Perplexity 輸出時「有格式可循」
```

**步驟**：

```
Step 1: 設計 JSON 結構（簡化版）
  → 目標：CORE_GATE 的輸出轉 JSON

Step 2: 在 GPT 上測試一個案例
  → 提示詞：「把這個法律案件轉成 JSON」
  → 看格式對不對

Step 3: 製作「輸出 JSON 範本」
  → 一個案例 + 一個複雜案例

Step 4: 更新 CORE_GATE 文檔
  → 在「階段 2 輸出」加 JSON 結構說明
```

**產出物**（簡化版 JSON）：

```json
{
  "case_id": "CASE_20260116_001",
  "fact_status": "VERIFIED",
  "domain": "刑法",
  "risk_level": "A_CLASS",
  "warnings": [
    "死刑罪名，需人工複核"
  ],
  "verified_facts": {
    "who": "行為人 A",
    "when": "2025-12-15",
    "where": "台灣台北市",
    "what": "轉帳 500 萬元"
  },
  "unverified_facts": {
    "what": "轉帳目的"
  },
  "next_action": "需人工複核"
}
```

**實施時間**：2～3 小時  
**收益**：✅ 為後續所有系統銜接打基礎

#### 【第 4～5 天】② Alert Log（簡化版）✅ 快速追蹤

**工作量**：⭐⭐ 1～2 天

**簡化版策略**：

```
【完整版 Alert Log】
  需要：
  - 每個決策點都記錄
  - 時間戳記
  - 複雜的結構化日誌
  - 後端數據庫

【簡化版】
  只記錄「關鍵決策點」
  ├─ 缺失事實偵測 → 記錄
  ├─ 回溯發生 → 記錄
  ├─ 高風險標註 → 記錄
  └─ 其他決策點 → 跳過

  用「簡單文本日誌」就行
  不需要複雜資料庫
```

**步驟**：

```
Step 1: 定義「3 個必記錄的時刻」
  ├─ 事實不明發生 → [時間] 缺失項目
  ├─ 回溯觸發 → [時間] 回溯原因
  └─ 高風險標註 → [時間] 風險等級

Step 2: 製作「簡單範本」
  → 每個案件一份 .txt 日誌

Step 3: 在 Perplexity 說明如何記錄
  → 在 MODE_QC_v2.0.md 加「日誌記錄」段落

Step 4: 測試 1～2 個案例
  → 看日誌記錄有沒有用
```

**產出物**（簡化版日誌）：

```
【案件 CASE_20260116_001 — 決策日誌】

時刻    事件類型        描述
────────────────────────────────────
18:37:00 缺失事實偵測    WHO 不明，進入容忍機制
18:37:10 法域判斷        刑法 + 行政法，判斷為複域案件
18:37:15 高風險標註      A 類（死刑罪名）需人工複核
18:37:20 最終狀態        待人工複核

【決策軌跡】
✓ 資訊部分不明但可進 Stage 2
✓ 複域案件已識別
✓ 已推送至人工複核隊列
```

**實施時間**：1～2 天  
**收益**：✅ 案件追蹤清晰化

---

### **第 2～3 週：深度優化（一人能做的）**

#### 【第 6～8 天】⑤ Human-in-Loop（簡化版）✅ 必做

**工作量**：⭐⭐ 2～3 天

**簡化版設計**：

```
【完整版】
  需要：
  - 專門的複核 UI 介面
  - 複核人員帳戶管理
  - 複核簽核流程
  - 與系統深度整合

【簡化版】
  一人操作：
  - 在 Perplexity 的「最終輸出」加一行
  - 「⚠️ A 類高風險，需人工複核」
  - 用「複核清單.md」文件追蹤
  - GPT 上模擬複核流程
```

**步驟**：

```
Step 1: 定義「複核觸發條件」
  ├─ A 類風險 → 強制複核
  ├─ B 類風險 → 需考慮複核
  └─ C 類風險 → 標記提醒

Step 2: 製作「複核清單模板」
  → 簡單 .md 文件，手工跟蹤
  ├─ 案件 ID
  ├─ 風險等級
  ├─ 複核要點
  ├─ 複核人決定
  └─ 時間戳記

Step 3: 在 Perplexity CORE_GATE 說明
  → 加「第 6 步：人工複核標記」

Step 4: 在 GPT 上模擬複核流程
  → 提示詞：「假設你是複核人員，這個案件需要複核嗎？」
  → 看 GPT 的複核邏輯
```

**產出物**（簡化版複核清單）：

```markdown
# 【複核清單】— 2026-01-16

## 待複核案件

### CASE_20260116_001
- 風險等級：A 類（死刑罪名）
- 複核要點：
  ✓ 行為人身份確認
  ✓ 主觀故意判斷
  ✓ 法域確認（刑法 vs 行政法）
- 複核人：[待指派]
- 複核期限：2026-01-17 10:00
- 決定：[待複核]

## 已複核案件

### CASE_20260115_001
- 風險等級：NORMAL
- 決定：✅ 通過，不需複核
- 複核人：系統自動
- 複核時間：2026-01-16 15:30
```

**實施時間**：2～3 天  
**收益**：✅ 高風險案件的安全把關

#### 【第 9～14 天】③ 多域橋接（部分實現）🟡 分階段

**工作量**：⭐⭐ 2～3 天（第一階段）

**為什麼分階段？**

```
完整版多域評估很複雜（需要 1 周以上）
但你可以先做「部分版本」：

【第一階段 — 簡化版】
  只支援「兩域並行」
  ├─ 刑法 + 行政法（環保、食安常見）
  ├─ 民法 + 刑法（侵權+刑責常見）
  └─ 其他複域案件 → 標記為「需人工判斷」

【第二階段 — 完整版】
  支援「三域甚至多域」
  └─ 後期再做（可能 2 月中旬）
```

**步驟**（第一階段）：

```
Step 1: 定義「常見複域模式」
  ├─ 環保污染：刑法 + 行政法
  ├─ 食安違規：刑法 + 行政法
  ├─ 詐欺/侵權：刑法 + 民法
  └─ 其他 → 標記「複雜案件」

Step 2: 製作「簡單決策樹」
  → 如果是「已知的複域模式」
     → 自動啟動兩個評估軌道
  → 否則
     → 標記為「需人工判斷」

Step 3: 在 CORE_GATE 中加「複域判斷」
  → 在「法律領域選擇樹」後面加一步

Step 4: 測試 2～3 個複域案件
  → 在 GPT 上跑一遍，看流程對不對
```

**產出物**（簡化版多域判斷）：

```markdown
## 【複域案件判斷】— v1.2.0 簡化版

### 已支援的複域模式

#### 模式 1：刑法 + 行政法（環保/食安）
- 觸發條件：違反《環保法》或《食安法》 + 有刑事責任
- 評估方式：
  ├─ 刑法軌道：是否構成犯罪？
  ├─ 行政軌道：是否違反行政規定？
  └─ 銜接檢查：兩個責任是否衝突？

#### 模式 2：民法 + 刑法（侵權）
- 觸發條件：同時涉及民事侵權 + 刑事責任
- 評估方式：
  ├─ 民法軌道：侵權成立嗎？
  ├─ 刑法軌道：刑責成立嗎？
  └─ 銜接檢查：損害賠償是否重複？

### 暫不支援的複域模式

❌ 三域以上的複雜案件
   → 標記為「需人工判斷」
   → 轉入高風險複核流程

【簡化版優勢】
✅ 一人可維護
✅ 清晰易懂
✅ 先覆蓋 80% 的複域案件
✅ 其餘 20% 靠人工複核補救
```

**實施時間**：2～3 天（第一階段）  
**收益**：✅ 基本複域案件可初步評估

## 💡 一人團隊的「聰明捷徑」

### 你可以做的取巧優化

```
【用 GPT 的能力代替工程能力】

1. JSON 標準化
   → 不用寫程式，用 GPT 生成範本
   → 用「聲明式」方式定義，而不是程式碼

2. Alert Log
   → 不用後端資料庫
   → 用簡單 .md 文件 + 手工記錄
   → 一樣可追蹤

3. 複核流程
   → 不用複雜的簽核系統
   → 用簡單清單 + GPT 模擬複核邏輯
   → 真正需要時再人工複核

【核心想法】
用「聲明」代替「實現」
```
