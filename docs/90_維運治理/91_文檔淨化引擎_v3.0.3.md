# 92_文檔淨化引擎_v3.0.3

## 核心目的

把智研文件整理成可被 AI 代理穩定載入的結構。淨化重點不是美化文字，而是降低重複規則、舊版混入、連結斷裂、模式漂移與引用政策衝突。

## 淨化順序

1. 判斷文件用途：入口、核心控制、模式引用、模組人格、封存參考、維運治理。
2. 若文件是舊版、簡版或重複規格，移入 `80_封存參考`。
3. 若文件是主流程規則，放入 `10_核心控制層`、`20_模式與引用層` 或 `40_模組與人格層`。
4. 修正檔名型 `[[連結]]`，確保可解析到現有 Markdown 檔。
5. 概念詞 `[[連結]]` 可保留，不強制建檔。
6. 每個 Markdown 第一行必須是標準 H1。
7. 更新 `90_檔案清單表_v2026.2.28` 與 `96_匯出對照_EXPORT_MAP_v1.0.0`。

## 分類規則

| 類型 | 目標資料夾 | 判斷標準 |
|---|---|---|
| 入口、README | `00_入口與總覽` | 使用者或代理第一步會讀 |
| 主人格、啟動、閘門、空間核心 | `10_核心控制層` | 決定全域規則或流程 |
| REPORT / RESEARCH / QC / 引用政策 | `20_模式與引用層` | 決定輸出格式與引用格式 |
| 功能模組與人格 | `40_模組與人格層` | 依任務啟用 |
| 舊版、簡版、參考 | `80_封存參考` | 不應進主流程 |
| 清單、測試、治理 | `90_維運治理` | 維護時使用 |

## 常見問題與解法

| 問題 | 解法 |
|---|---|
| 同一功能有主檔與簡版 | 主檔留在流程層，簡版移入 `80_封存參考` |
| 檔案名稱已更新但內文仍指舊名 | 只修檔名型連結，不改概念詞 |
| 代理把 deprecated 當主規則 | 封存資料夾不進主流程，匯出對照明確排除 |
| H1 與檔名不一致 | H1 改成目前檔名，方便搜尋與讀取 |
| 空檔被其他檔案引用 | 補最小骨架，標明待補，不虛構完整規格 |

## 淨化完成條件

- 根目錄無散落 Markdown。
- 全部 Markdown 都在分類資料夾內。
- 檔名型 `[[連結]]` 未解析數為 0。
- 空檔數為 0。
- 重複前綴不影響同資料夾排序。
- `80_封存參考` 不出現在主流程匯出包。
